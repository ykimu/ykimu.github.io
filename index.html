ソフトウェアが抱える技術的負債を、データから解き明かす Understanding the technical debt that software carries, from data

木村 祐太 / Yuta Kimura (Yuta Kimura)
大阪公立大学工業高等専門学校 助教Assistant Professor, Osaka Metropolitan University College of Technology

ソフトウェア工学・ソフトウェア保守を専門に、開発リポジトリのデータを分析し、保守を難しくする「技術的負債」の検出と特徴づけに取り組んでいます。 I work in software engineering and software maintenance, mining development repositories to detect and characterize the technical debt that makes software harder to maintain.

研究についてAbout the research

「後で直す」が積み重なる前にBefore "we'll fix it later" piles up

ソフトウェアは長く使われるほど保守の費用が膨らみます。その一因が技術的負債（Technical Debt）、すなわちその場しのぎの設計や実装が後の変更を高くつかせる現象です。私は、GitHubなどの開発リポジトリや課題管理システムに蓄積された大量のデータを分析し、開発者が実際に「負債」と認識した箇所を、ソフトウェアメトリクスや機械学習を用いてどこまで検出・分類・特徴づけできるかを研究しています。 The longer software lives, the more it costs to maintain, and one driver is technical debt: expedient design or implementation that makes future changes costlier. I mine the large volumes of data accumulated in development repositories and issue trackers to study how far we can detect, classify, and characterize the code that developers themselves recognize as debt, using software metrics and machine learning.

近年は、技術的負債の返済とリファクタリングの関係、自動生成パッチがコーディング規約に与える影響、開発者によるAI・大規模言語モデルの利用など、実務に根ざしたテーマにも広げています。 More recently I have extended this to practically grounded topics such as the relationship between debt repayment and refactoring, the impact of auto-generated patches on coding standards, and how developers use AI and large language models in practice.

Technical Debt × Software Metrics

課題管理システムで報告された技術的負債の、メトリクスによる分類と特徴づけClassifying & characterizing technical debt reported in issue tracking systems using software metrics

GitHub上のJava 140プロジェクト・約7.3万ファイルを対象に、77のソフトウェアメトリクスで「負債と報告されたファイル」をどこまで捉えられるかを検証。5つの開発履歴メトリクスが72のソースコード指標に匹敵しました。 Across 140 Java projects on GitHub (~73k files), 77 metrics are tested for capturing files reported as debt. Just 5 development-history metrics rival 72 source-code metrics.

詳細ページを見るRead the highlight ›
分類の正解率Classification accuracy
全指標 (77)All (77) 0.827
ファイル (5)File (5) 0.822
コード (72)Source (72) 0.765

開発履歴だけの5指標が72のソースコード指標に匹敵（基準 0.644）。5 history metrics rival 72 source-code metrics (baseline 0.644).

APR × Coding Standards

自動生成されたパッチがコーディング規約に与える影響の分析The impact of automatically-generated patches on coding standards

Defects4Jの3プロジェクトに32のAPRツールのパッチを適用し、Checkstyleで規約違反を計測。テストに通っても、生成コードの「読みやすさ」がどれだけ損なわれるかを定量化しました。 Applying 32 APR tools to three Defects4J projects and measuring violations with Checkstyle, this study quantifies how much auto-generated code sacrifices readability even when tests pass.

研究紹介を見るRead the highlight ›
コーディング規約違反の数Coding-standard violations
修正前Before 100
APR適用後After APR ≈140

規約違反が約40%増加。うち約9〜10%は可読性に影響。About 40% more violations; ~9-10% affect readability.

Technical Debt × Refactoring

課題管理システムにおける技術的負債の返済とリファクタリングの関係の調査The relationship between technical-debt repayment and refactoring

GitHubの139 Javaプロジェクトを対象に、TD-Issueに紐づく返済コミットをRefactoringMinerで解析。負債の「返済」が実際どれだけリファクタリングを伴うのかを明らかにしました。 Analyzing repayment commits linked to TD-Issues across 139 Java projects with RefactoringMiner, this study reveals how far debt "repayment" actually involves refactoring.

研究紹介を見るRead the highlight ›
リファクタリングの関与Refactoring involvement
返済コミットCommits 59.6%
変更ファイルFiles 32.4%

返済コミットの過半数がリファクタリングを含み、複数操作を組み合わせる。Most repayment commits include refactoring, often combining operations.

SATD-Issue Classification

技術的負債に関連する課題票分類手法の構築Building an issue classification model on technical debt

開発者が課題票で明示した負債（SATD-Issue）を自動で見つける分類モデルを構築。データ不足に阻まれてきたSATD-Issue研究の基盤を広げます。 A classifier that automatically finds debt developers flag as issue tickets (SATD-Issue), widening a research area long blocked by data scarcity.

研究紹介を見るRead the highlight ›
分類性能（最良・vscode）Performance (best · vscode)
Precision 0.995
Recall 0.903
F1 0.947

TF-IDF＋SVMで高精度に分類（vscode）。High accuracy with TF-IDF + SVM (vscode).
